‚ö†Ô∏è Vulnerability 1: Clickjacking (Missing X-Frame-Options)

Description

The web application does not implement clickjacking protection. Pages can be embedded inside an iframe on a malicious website, allowing attackers to trick users into clicking hidden UI elements.

Steps to Reproduce

<iframe src="https://www.[REDACTED].com/login" style="width:100%; height:100%; opacity:0.1;"></iframe>

Create an HTML file with the above iframe. When loaded, the target site appears inside the iframe, allowing overlay attacks.

Impact

An attacker can trick users into performing sensitive actions (login, profile updates, form submissions) without their knowledge, leading to account compromise or unauthorized transactions.

Recommendation

Add the following headers:

X-Frame-Options: DENY
Content-Security-Policy: frame-ancestors 'none';

‚ö†Ô∏è Vulnerability 2: Missing Rate Limiting (Authentication Endpoint)

Description

The authentication endpoint does not implement rate limiting, account lockout, or CAPTCHA mechanisms. An attacker can send unlimited login attempts without restrictions.

Steps to Reproduce

  • Capture login request in Burp Suite
  • Send to Intruder with password field as payload position
  • Load password list (e.g., rockyou.txt)
  • Start attack - 4,000+ requests sent within minutes

Impact

Attackers can brute-force accounts using automated tools. High risk of credential stuffing and account takeover affecting all users.

Recommendation

  • Implement rate limiting (max 5 attempts per minute per IP)
  • Temporary IP blocking after repeated failures
  • CAPTCHA after multiple failed attempts
  • Account lockout after threshold of failed logins

‚ö†Ô∏è Vulnerability 3: Weak Password Policy

Description

The application allows users to create passwords that are too short or lack complexity. No enforced rules for minimum length, uppercase/lowercase, numbers, or special characters.

Steps to Reproduce

  • Navigate to signup page
  • Enter simple password like "123456" or "password"
  • No password strength validation messages appear
  • Account successfully created with weak password

Impact

Attackers can easily guess user passwords using brute force or credential stuffing. Accounts are at high risk of compromise, potentially exposing sensitive data.

Recommendation

  • Enforce minimum 8-12 characters
  • Require uppercase, lowercase, numbers, and symbols
  • Implement password strength meter
  • Consider multi-factor authentication (MFA)

‚ö†Ô∏è Vulnerability 4: Insufficient Security Logging & Monitoring

Description

The web application does not implement adequate logging and monitoring for security-relevant events. Critical actions such as failed login attempts, privilege changes, and sensitive API requests are not logged or monitored.

Steps to Reproduce

  • Using Burp Intruder, send 4,000+ login attempts
  • All requests reach server without being blocked
  • No alerts triggered, no IP blocking, no rate limiting

Impact

Attackers can perform brute force attacks undetected. Malicious activity may go unnoticed, delaying incident response and increasing risk of account compromise and data breach.

Recommendation

  • Log all security-relevant events (login attempts, privilege changes, API access)
  • Implement monitoring and alerting for suspicious activity
  • Use centralized logging system (ELK Stack, Splunk)
  • Implement automated defense mechanisms (rate limiting, IP blocks)

üö® CRITICAL: Insecure Firebase Security Rules

Description

The Firebase backend was configured with permissive Security Rules granting public read access to sensitive data stores. While Firebase API keys are designed to be publicly accessible in client-side code, they must be paired with restrictive server-side Security Rules that enforce authentication.

Root Cause: Firebase Security Rules configured to allow unauthenticated access (allow read: if true; or equivalent).

Steps to Reproduce

  • Login to application with valid credentials
  • Use Burp Suite to intercept authentication request
  • Observe Firebase API key in POST request URL parameters
  • Extract API key and test against Firebase services
  • Result: Successfully accessed Firestore database and Cloud Storage using only the intercepted API key

Impact - CRITICAL (CVSS 8.6)

An attacker can:

  • Enumerate and download all user-uploaded images/files
  • Access internal business media and assets
  • Identify user data through file names, metadata, or image contents
  • If write rules are also misconfigured, host malware in publicly accessible buckets

This compromises confidentiality and violates user privacy and regulatory requirements.

Immediate Recommendation

// Firebase Storage Security Rules - FIX IMMEDIATELY
rules_version = '2';
service firebase.storage {
match /b/{bucket}/o {
match /{allPaths=**} {
allow read, write: if request.auth != null;
}
}
}

// Firestore Security Rules
service cloud.firestore {
match /databases/{database}/documents {
match /{document=**} {
allow read, write: if request.auth != null;
}
}
}
  • Rotate tokens for all existing files
  • Review all public files - only explicitly mark necessary folders with "allow read"
  • Do not rely on API keys for security ‚Äî they are public by design

üìù Conclusion

This web application vulnerability assessment revealed multiple security misconfigurations ranging from missing security headers to a critical Firebase misconfiguration exposing user data. The most severe finding was the insecure Firebase Security Rules allowing unauthenticated access to user files and database contents.

The application demonstrates strong foundational security with Vercel hosting, HTTPS enforcement, and HSTS, but fails to implement critical application-layer protections including rate limiting, proper logging, and security headers.

‚ö†Ô∏è Critical Takeaway: Firebase API keys are public by design and must never be trusted for security. Proper Security Rules enforcing authentication are essential. Additionally, rate limiting and security headers are basic protections that should be implemented on all production web applications.
